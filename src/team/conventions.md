# 协作规范

## 🗣️ 沟通约定

### 日常交流
- **微信群**：用于即时讨论、问题求助
- **论坛**：重要通知、技术分享、会议纪要归档
- **邮件**：正式申请、对外联络（可选）

### 称呼与态度
- 互称昵称或用户名，随意自然
- 技术讨论对事不对人
- 遇到问题先查文档，再提问

## 💻 开发规范

### 代码管理
```
分支命名：
- 功能：feature/简短描述
- 修复：fix/问题描述
- 文档：docs/更新内容

提交信息格式：
类型(范围): 描述
例：feat(auth): 增加登录验证
```

### 项目结构
新项目应在根目录包含：
- `README.md` - 项目说明、如何运行
- `docs/` - 详细文档（如有）
- `.gitignore` - 排除不必要的文件

## 📅 会议制度

### 周会（F组组织）
- **时间**：每周一次，具体时间群内投票确定
- **时长**：不超过1小时
- **内容**：
  1. 各项目进度同步（每人1-2分钟）
  2. 新提案讨论（如有）
  3. 问题协调
- **纪要**：由当次会议指定成员记录，发至论坛

### 项目会
- 按需召开，由项目负责人组织
- 会前明确议题，控制时长

## 🎯 质量要求

### 文档
- 公开项目必须有`README`
- 复杂功能应有使用说明
- 关键决策记录在案

### 代码
- 提交前自测基本功能
- PR应有描述和测试说明
- 鼓励代码审查，但不是强制

## 🔄 加入与贡献

### 新成员引导（D组负责）
1. 介绍群、论坛、GitHub组织
2. 指引查看现有项目
3. 协助选择首个任务

### 贡献流程
```
发现兴趣 → 群内表达意向 → 认领任务 → 开始贡献
```
- 任何成员可直接在项目提Issue
- 开发任务向B组同步进度
- 设计/文案需求联系C组

## ⚠️ 注意事项

1. **尊重时间**：非紧急问题避免深夜@所有人
2. **主动沟通**：遇到阻塞及时提出，不必硬撑
3. **保持学习**：技术选型以适用为先，鼓励尝试
4. **享受过程**：这是兴趣团体，别让协作成为负担
